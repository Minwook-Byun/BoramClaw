#코딩 스타일 가이드 

이 문서는 프로젝트의 선호하는 코딩 패턴과 스타일 결정을 기록합니다. 이 문서를 수정할 대에는 가급적 간결하고 중복이 없도록 해야 합니다. 
# Coding Style Guide

**최소한의 출력 + 새로운 파일을 불필요하게 만들지 말 것** 

## 1) 목표
- 읽기 쉬운 코드가 우선이다.
- 동작하는 코드보다, 유지보수 가능한 코드가 더 중요하다.
- 필요 이상으로 복잡하게 만들지 않는다.

## 2) 기본 원칙
- 중복을 줄이고, 하나의 책임만 갖는 함수로 나눈다.
- 추측으로 코드를 쓰지 말고 입력/출력 계약을 명확히 한다.
- 로컬에서 빠르게 확인 가능한 최소 검증을 항상 수행한다.

## 3) Python 스타일
- PEP 8을 기본으로 따른다.
- 함수/변수명은 `snake_case`, 클래스명은 `PascalCase`를 사용한다.
- 타입 힌트를 기본으로 작성한다.
- import 순서는 표준 라이브러리 → 서드파티 → 로컬 모듈 순서로 정렬한다.
- 매직 넘버/문자열은 상수로 분리한다.

## 4) 함수 작성 규칙
- 한 함수는 한 가지 일만 하도록 유지한다.
- 함수 길이는 가능한 짧게 유지하고, 30~40줄을 넘기면 분리를 검토한다.
- 입력 파라미터가 많아지면 객체/구조체로 묶는 것을 우선 고려한다.
- 부작용(side effect)이 있는 함수는 이름으로 의도를 드러낸다.

## 5) 에러 처리
- 예외는 가능한 구체적으로 처리한다.
- `except Exception`은 최후의 수단으로만 사용한다.
- 에러 메시지는 원인 파악에 필요한 정보만 포함하고, 민감 정보는 노출하지 않는다.
- 실패 시 조용히 무시하지 말고, 호출자에게 명확히 전달한다.

## 6) I/O 및 로깅
- 사용자 출력은 최소화하고, 필요한 정보만 출력한다.
- 디버그 출력은 기본 비활성화 상태로 유지한다.
- 외부 API 호출, 파일 I/O 등은 실패 가능성을 항상 고려한다.

## 7) 성능
- 먼저 단순하게 구현하고, 병목이 확인되면 최적화한다.
- 불필요한 객체 생성/복사를 줄인다.
- 반복 호출되는 네트워크/리소스는 재사용 가능한 연결을 우선한다.

## 8) 테스트와 검증
- 변경 시 최소한의 실행 검증(문법 검사/핵심 경로 실행)은 반드시 수행한다.
- 버그 수정은 재현 케이스를 남기고, 가능하면 테스트로 고정한다.
- 테스트는 빠르고 독립적으로 실행 가능해야 한다.

## 9) 리뷰 체크리스트
- 이름만 보고도 역할이 이해되는가?
- 예외 처리 경로가 실제로 안전한가?
- 불필요한 출력/주석/죽은 코드가 남아 있지 않은가?
- 요구사항을 벗어난 과도한 구현이 없는가?
